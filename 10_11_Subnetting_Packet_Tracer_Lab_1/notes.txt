    Exact IP Address Breakdown
    Here is the device-by-device list of what you need to configure. The subnet mask for every device on this list will be 255.255.255.192.

    Subnet 1: Site 1 LAN (192.168.1.0/26)
    Usable Range: 192.168.1.1 - 192.168.1.62

    Router 1 (LAN Interface): 192.168.1.1 (Standard practice is the first usable IP)

    Switch 1 (Management IP): 192.168.1.61 (Rule: Second last IP -> 62 - 1)

    DHCP Server 1: 192.168.1.60 (Rule: Third last IP -> 62 - 2)

    PC A: Receives its IP automatically from the DHCP Server. It will likely get 192.168.1.2 (the first available IP in the DHCP pool).

    Subnet 2: Link between R1 and Internet Router (192.168.1.64/26)
    Usable Range: 192.168.1.65 - 192.168.1.126

    Router 1 (WAN Interface): 192.168.1.65 (First usable IP)

    Internet Router (Interface facing R1): 192.168.1.66 (Second usable IP)

    Subnet 3: Site 2 LAN (192.168.1.128/26)
    Usable Range: 192.168.1.129 - 192.168.1.190

    Router 2 (LAN Interface): 192.168.1.129 (First usable IP)

    Switch 2 (Management IP): 192.168.1.189 (Rule: Second last IP -> 190 - 1)

    DHCP Server 2: 192.168.1.188 (Rule: Third last IP -> 190 - 2)

    PC B: Receives its IP automatically from the DHCP Server. It will likely get 192.168.1.130.

    Subnet 4: Link between R2 and Internet Router (192.168.1.192/26)

    Router 2 (WAN Interface): 192.168.1.193 (First usable IP)

    Internet Router (Interface facing R2): 192.168.1.194 (Second usable IP)

    For Step 5:
        1. The PC Configuration (DNS is Mandatory)
        As we discussed earlier, computers do not understand domain names. If PC C opens a browser and types facebook.com, the very first thing it does is try to translate that name into an IP address using a DNS server.

        Because the instructions for Site 3 said to manually configure all devices, you do not have a DHCP server handing out this information. You must go to PC C's IPv4 configuration screen and type it in yourself:

        IP Address: 192.168.1.65

        Subnet Mask: 255.255.255.240 (The /28 mask)

        Default Gateway: 192.168.1.78 (R3's LAN IP)

        DNS Server: 8.8.8.8 (Or whatever DNS server IP your lab environment dictates)

        If you skip the DNS server entry, the ping to an IP address might work, but typing the URL in the browser will fail instantly.

        3. Static Return Routes (Getting Traffic BACK)
        This is the step that trips up almost every CCNA student!

        Let's assume the packet successfully reached Facebook, and Facebook is sending the webpage back to PC C (192.168.1.65). The packet arrives at the Internet Router.

        The Internet Router looks at the destination IP (192.168.1.65). It checks its routing table. The Internet Router only knows about the three /30 serial links it is directly connected to. It has no idea that the 192.168.1.64/28 LAN exists behind R3. Because it doesn't know where to send it, the Internet Router drops the packet, and the connection times out.

        You must teach the Internet Router how to reach your internal LANs using Static Routes.

        4. Network Address Translation (NAT)
        In a simulated lab environment (like basic Packet Tracer exercises), Step 3 is usually enough to make the ping succeed.

        However, in the real world (and in advanced CCNA labs), the ISP will completely ignore those private 192.168.1.x addresses. You would need to configure NAT Overload (PAT) on R1, R2, and R3 to translate the private LAN IPs into their public serial interface IPs before the traffic hits the Internet Router.

        If your lab instructions specifically mentioned configuring NAT earlier, that must be active. If not, the routing setup in Step 2 and Step 3 is exactly what your instructor is looking for.
