Comparision:
    - If we send packets through IP from one device to another, then there is no guarantee that packets reach the destination in the same sequence through which it was sent
    - Need to ensure reordering of packets
    - TCP/UDP are part of the layer 4, i.e., the transport layer
    - UDP is connectionless like IP and does not guarantee the delivery of packets
    - UDP is simple and scalable but is unreliable
    - TCP provides delivery acknowledgement and reliability and is connetion oriented
    - Origin to Destination, Call Initiated -> Destination responds to Source -> Source Responds to the destination
    - In TCP : Initiate Call, Acknowledge Call, Indicate Speaking and then communication is established
    - Socket is a combination of IP address of a host and the port number which is used and the trasport protocol which is being used : TCP/UDP
    - TCP/UDP allows session multiplexing, which is a scenario when a single host with single IP address is able to communicate with multiple server or multiple sessions and have multiple sessions simultaneously
    - A session is created when a source host is created when a source host needs to send information or data to the destination host
    - The sessions are created and controlled with the network application which contains the functionality of OSI layer 5 through 7.
    - Information may be segmented or chunked for the transmission across a physical medium
    - Maximum Segment Size (MSS) is the largest amount of data in bytes that TCP is willing to send in a single segment and should be small enough
    - TCP support Maximum Segment Size and Maximum Transmission Unit discovery
    - MSS (actual data) is only a transport 4 layer phenomenon while MTU is considered for all the layers and are considered in bytes
    - MSS = MTU - (IP Header + TCP Header)
    - Flow control is a phenomenon in TCP to avoid sending data too quickly since if sender sends data faster than receiver can handle, receiver drops the data and require retransmission
    - TCP have sliding window to control the flow of data. Windowing allows the recieving computers on how much data it is able to receive before transmitting an acknowledgement to the sending computer
    - UDP does not implement flow control unlike TCP and it relies on higher layer protocols to implement flow controls
    - UDP has no data recovery protocol and relies on higher order protocol. TCP 3 Way Handshake:
    - Users of TCP must establish a connection oriented session with a peer System
    - Host A initiating the session will set the SYN Flag in the TCP header of the initial segment while sending to Host B (Step 1)
    - Host A also chose an initial SYN number of SEQ=100 (example is 100)
    - This is used to start the handshake process
    - The synchronization process also specifies the number of port, the sender wants to connect
    - Once SYN is recieved and accepted, HOST B sends SYN and ACK (101) flag set (Step 2)
    - Host B also sets an initial sequence number of SEQ=300 (ex. is 300) to indicate the next byte of data it expects to acknowledge
    - Once SYN has has been received back by the HOST A, it signifies that the session has been established and the CTL field has been set to ACK and the the ACK field set to 301
    - SEQ number has now been set to 101. Then the SYN flag is unset and it signifies that the 3 way handshake has completed successfully
    - Note Initial sequence number can differ and ACK indicates next sequence number that hosts expects to receive
    - Flow control prevents the issue of a sender overflowing the buffers of a receiver and is governed by the READY and NON READY flag

Windowing:
    - Window size is the number of data segments the sender is allowed to send before getting any acknowledgement from the receiver
    - With Window size of 1 (low window size), the sender cannot send the data until it gets the acknowledgement from the receiver
    - TCP allows for greater window sizes, i.e., more segments to be transmitted before receiving an acknowledgement
    - Choosing a greater window size can improve the throughput dramatically
    - Rather than using fixed windows, host tends to use sliding windows - where host initially starts with a small window size, while exponentially increasing the window size to guage the amount of data the receiver can receive and what the network can 
    - When a packet is dropped by the network, then the host will slow down
    - Congestion Window (CWND) is initially set to a very low value and then increases at an exponential rate
    - For every lost segment, the CWND is halved and through the Congestion Avoidance algorithm, it fixes the optimal window sizes
    - With sliding window, there is a dynamic negotiation of window size and that the window size may change dramatically during a session depending on what the receiver can process and what the network can handle
